# Apply Code Conventions

This skill is write-capable but does not create edit authority. If the user has not authorized a concrete Unity code change and target scope, stop before writing and ask for that scope.

1. Resolve the exact repository or worktree. Read every applicable `AGENTS.md`, `CLAUDE.md`, primary project router, directory-scoped instruction, safety rule, validation rule, and any remaining local convention before editing. Continue normally when local convention documents are absent.
2. Capture current Git state and preserve pre-existing changes. Retain the complete set of repository instructions, routed project documents, validation rules, and owner guides read during steps 1-5 for the `AFCC-DOC-001` completion gate. Do not move the work into another checkout or alter branches, worktrees, or index state through this skill.
3. Read the installed `com.actionfit.ai-codeconvention/AI_GUIDE.md` and `references/unity-code-authoring-rules.md`. Resolve the exact router selector through `AFCC-PRO-001`; use `portable-core` when absent and stop for duplicate or unknown selectors. Read `references/profiles/actionfit-unity.md` only when selected, and activate optional rules only with `AFCC-PRO-002` evidence.
4. Read `references/owner-routing.md` and every installed owner package guide relevant to an API being used. For serialized references, resolve the required `com.actionfit.referencebinding/AI_GUIDE.md` and use only that contract. If the owner is absent or its declared version cannot be resolved, stop before authoring serialized-reference code and report `Package/API Mismatch`; do not invent a fallback API. Use project architecture only to resolve concrete factual types, paths, owners, keys, and lifetimes; do not treat it as a duplicate general convention body.
5. Resolve conflicts through `AFCC-PRE-001`: higher project instructions and factual API owners win. Record the selected source and do not blend incompatible rules.
6. Before changing a public API, state store, event, query, command, save key/schema, asset key, asmdef, required serialized-reference rule, or resource lifetime, prepare the `AFCC-DEC-001` Decision Log. Stop for a material unresolved decision.
7. Implement only the authorized files and behavior. Preserve serialized values, GUIDs, public contracts, user changes, and existing architecture. Apply `AFCC-TRE-001`, `AFCC-PKG-001`, `AFCC-PCR-001`, `AFCC-PRT-001`, `AFCC-BND-001`, `AFCC-ANI-001`, and `AFCC-PKG-002` only when the authorized scope explicitly includes architecture or package boundaries and direct evidence identifies owners, dependency edges, binder/logic/animation ownership, package-neutral behavior, any qualifying external capability, and any product-owned composition root. Author the exact product marker pair only in an explicitly authorized product package root `AI_GUIDE.md` `Package Identity`, verify its actual ID against the sibling manifest, and never duplicate it in project documentation or infer it from installation. Do not treat an AI Refactor proposal as edit authority. A marker likewise creates no package-creation, migration, deletion, publication, or deployment authority. Do not introduce a bus, global registry, dependency, scripting symbol, framework, DI container, service locator, speculative interface, or migration merely because the package or profile describes one.
8. When `actionfit-unity` activates `AFCC-RFS-002`, `AFCC-RFS-005`, and `AFCC-SER-004`, keep `Refs`, `Assets`, and `Settings` roles separate, use private serialized backing fields with getter-only access, reject public setters, mutation methods, or runtime assignments of authored inputs, and move changing values into a separate runtime model. Require `[SerializeField, RequiredChildReference("ExactGameObjectName")]` on every newly created or deliberately revised supported mandatory `Refs` Component field and rely on the package-owned Editor pump instead of adding consumer `OnValidate` solely for ReferenceBinding. Preserve existing deprecated `AFCC-RFS-003` `RequiredReference` plus `AutoWireChild` pairs and deprecated `AFCC-RFS-004` consumer-enqueued integrated declarations without bulk migration. Untouched legacy fields are not bulk-migrated. Identify the intended selector candidate from an existing valid serialized reference or an explicit user decision. Never select the first duplicate or omit the attribute: rename only non-selected duplicates inside an explicitly authorized asset scope, use deterministic collision-free temporary names derived from stable hierarchy order, report every old/new path, inspect runtime-name and Animation-binding impact plus the targeted serialized diff, and verify reload without reference loss. Stop when candidate evidence, asset authority, or validation is missing.
9. Request separate approval before any destructive or sensitive asset, scene, prefab, ScriptableObject, ProjectSettings, serialization, migration, credential, build, release, or external-system change not already named by the user.
10. Run the smallest validation that proves the changed contract, then add compile, EditMode, PlayMode, asset, persistence, capability-negative, or manual QA evidence in proportion to risk. Inspect the final targeted diff and remove only traces introduced by this work.
11. Apply `AFCC-DOC-001` before reporting completion. Re-read completely the primary router, every governing document retained in step 2, every relevant installed owner guide, and every additional document implicated by the final diff. Compare their behavior, ownership, public surface, validation path, and risks with the implementation. Update stale documentation only inside the authorized scope; otherwise report the mismatch as a blocker. Run every repository-declared documentation or package validator and record each exact command and result. A validator pass never replaces semantic re-reading. Report unrelated failures without fixing their owners, and report the absence of a declared validator instead of inventing a native final-response hook.
12. Report the selected profile, activated capability rules including `AFCC-RFS-005` and `AFCC-DOC-001` when applicable, owner guide paths, changed files, ownership/state/communication choices, the Decision Log when triggered, documentation re-read scope, validator results, unverified behavior, and remaining risks. Do not issue a normal completion report when step 11 is incomplete.

This skill does not own Jira, task selection, branch or worktree creation, commits, pushes, pull requests, package publication, repository creation, tags, catalog writes, deployment, production operations, or global/home skill installation. Use the repository's dedicated workflow and separate authorization for those operations.
